# Vercel 部署与同步说明

## 当前结论

项目已经接通以下链路：

- 本地工作区：`E:\zhijiang`
- GitHub 仓库：`weichipan/zhijiangdaily`
- Vercel 项目：`zhijiangdaily`
- 正式域名：`zhijiangdaily.com` / `www.zhijiangdaily.com`

这意味着后续更新前端时，不需要再依赖“本地手工上传”。
标准路径应该是：

**本地改动 -> Git 提交 -> 推送到 GitHub -> Vercel 自动部署**

## 推荐工作流

### 1. 本地验证

先在本地完成开发和验证：

- `pnpm dev`
- 浏览器检查页面效果
- `next build`

`localhost` 只用于开发阶段确认改动，不是最终同步手段。

### 2. 预览部署

如果改动较大，先走预览分支。

推荐分支命名：

- `preview/<topic>`

流程：

1. 在本地创建预览分支
2. 只提交与本次改动直接相关的文件
3. 推送到 GitHub
4. 等待 Vercel 为该分支生成 preview 部署

适用场景：

- 首页改版
- 视觉重构
- 大范围样式调整
- 不想直接影响正式站时

### 3. 正式部署

当预览确认没问题后，把预览分支合并到 `main`。

只要 Vercel 项目把 `main` 设为生产分支，合并完成后就会自动重新部署正式站。

也就是说：

**合并到 `main` = 同步到主网站**

## 本项目的实际约定

### 正式站更新

- 生产分支：`main`
- 生产域名：`www.zhijiangdaily.com`
- 主域名 `zhijiangdaily.com` 应跳转到 `www`

### 预览站更新

- 使用 GitHub 分支触发 Vercel preview
- 不再优先依赖本地 CLI 或临时上传脚本

### 什么情况下还需要 localhost

只有这些情况仍然优先本地看：

- 新 UI 还没确认
- 数据结构刚改完，需要先本地验证
- 不确定会不会影响正式站
- 正在开发中，还不值得触发远程部署

## 为什么之前会反复看 localhost

因为当时需要先确认三件事：

1. 本地代码是否真的可运行
2. 首页改版是否先在浏览器中成立
3. 当前工作区是否已经和 GitHub / Vercel 的自动部署链路打通

在链路没确认前，直接假设“推了就能上站”风险更高。

现在这条链路已经验证通过，后续就应该优先使用 Git -> Vercel 自动部署。

## 后续操作准则

### 小改动

- 本地验证
- 在当前工作分支提交
- 合并到 `main`

### 大改动

- 本地验证
- 推 `preview/<topic>`
- 等 Vercel preview
- 确认无误后再合并到 `main`

### 不要做的事

- 不要把 `.next/`、`node_modules/`、临时打包目录提交到仓库
- 不要把只用于临时预览的目录作为长期发布方式
- 不要把并行会话里不相关的文档改动顺手混进生产提交

## 环境变量

生产环境至少需要：

- `ADMIN_PASSWORD`

如果线上要启用正式可写数据持久化，还需要：

- `KV_REST_API_URL`
- `KV_REST_API_TOKEN`

## 发布前检查

发布到 `main` 前至少确认：

- 首页能正常打开
- 关键图片能正常加载
- `/daily/[date]` 页面不报错
- `/admin/login` 可访问
- `next build` 已通过

## 发布后检查

合并到 `main` 后检查：

- `www.zhijiangdaily.com` 是否更新到最新页面
- 关键图片是否正常显示
- 后台是否还能登录
- 是否出现乱码、布局错位或资源 404
